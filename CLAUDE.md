# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A Django project for "Common Ground", a fictional shared counselling practice, built as a portfolio piece. It is a marketing site (homepage + four service pages + a counsellors directory) plus a real database-backed booking flow. All counsellor/topic/service content now lives in real DB tables (`ServiceCategory`, `Topic`, `Counsellor`, `CounsellorWorkingHours`, `CounsellorSpecialty` — see Architecture), editable via Django admin; `core/data.py` is a thin query layer over them, not static Python anymore. `Booking` and `CalendarAccount` are the only models still keyed by a plain `counsellor_slug` string rather than a FK (see the `core/models.py` bullet for why).

There is no `Pipfile`/`pyproject.toml` in the repo, but there is now a `requirements.txt` pinning `Django==6.0.6` and `Pillow==12.3.0` (Pillow is required by `Counsellor.photo`, an `ImageField`) — install with `pip install -r requirements.txt` before running anything; Django is **not** installed in the default environment.

## Commands

Run all commands from the repo root (where `manage.py` lives):

```bash
python manage.py runserver          # dev server at http://127.0.0.1:8000/
python manage.py migrate            # apply migrations — creates the Booking/CalendarAccount tables plus auth/admin/sessions
python manage.py makemigrations     # generate migrations after editing core/models.py
python manage.py test               # run the test suite (core/tests.py)
python manage.py test core.tests.SomeTestCase.test_method  # run a single test
python manage.py createsuperuser    # create an admin user for /admin/ — manage Bookings, and now Counsellor profiles/topics/services too
```

There is no linter or formatter configured in this repo.

## Architecture

Standard two-part Django layout: a project package for site-wide config, and one app for actual functionality.

- `ProjectCounsellingSite/` — the Django project package: `settings.py`, root `urls.py`, `wsgi.py`/`asgi.py`. `ROOT_URLCONF` delegates all non-admin routes to `core.urls`.
- `core/` — the (only) Django app.
  - `core/urls.py` — `''` → `views.index` (homepage), `'services/<slug:slug>/'` → `views.service_detail` (one route serving all four service pages, keyed by slug), `'counsellors/'` → `views.counsellors` (the full directory), `'book/'` → `views.book` (GET renders/POST submits the booking form), `'book/availability/'` → `views.booking_availability` (JSON slot lookup), `'book/<int:booking_id>/confirmed/'` → `views.book_confirmed`.
  - `core/data.py` — a DB accessor layer, **not** static content anymore: `get_topic_labels()` (slug → display label), `get_services()` (service slug → `{name, eyebrow, intro, topics: [slug, ...]}`), `get_counsellors()`/`get_counsellor_by_slug(slug)` (query `Counsellor` + related rows and reshape them into the exact same plain-dict shape the old static `COUNSELLORS` literal used to have — `slug`/`name`/`credentials`/`photo` (resolved to a final URL here, `c.photo.url` if a real upload exists else `static(c.photo_placeholder)` — this is the **only** place that resolves photo URLs; nothing downstream should call `static()` on a counsellor photo again)/`modes`/`languages`/`location`/`intro`/`bio`/`modalities`/`specialties` (topic-slug → weight dict, 2=primary/1=secondary, built from `CounsellorSpecialty` rows)/`fee_note`/`working_hours` (rebuilt from `CounsellorWorkingHours` rows into the same `{weekday_int: [("HH:MM","HH:MM"), ...]}` shape, Monday=0). Every function queries fresh on each call (never cached at import time) so admin edits appear immediately without a restart. Because the dict shape is unchanged, `views.py`/`forms.py`/`scheduling.py`/templates/JS needed only mechanical call-site updates (static literal access → function call) — none of them had to change what they do with the data.
  - `core/views.py` — `index`, `service_detail`, `counsellors` (`_resolved_counsellor` helper adds `specialty_labels` for the directory view; photo URLs already arrive pre-resolved from `core/data.py`, so views.py never calls `static()` itself), plus `book` (renders `booking.html`; on POST validates via `BookingForm`, creates a `Booking` inside `transaction.atomic()`, catches `IntegrityError` from the double-booking constraint and re-renders with a "slot just taken" error, then calls the stubbed `calendar_sync.create_calendar_event` and redirects to `book_confirmed`), `booking_availability` (GET, JSON: counsellor slug + date → available slot ISO timestamps, via `core/scheduling.get_available_slots`), `book_confirmed` (renders a static confirmation page for a given booking id).
  - `core/forms.py` — `BookingForm`, the project's first Django Form: `counsellor_slug`/`date`/`time`/`mode`/`client_name`/`client_email`, with `clean()` re-validating server-side that the counsellor exists, offers the requested mode, and that the requested slot is still in `get_available_slots(...)` — never trusts the client's fetch-rendered slot list.
  - `core/scheduling.py` — `get_available_slots(counsellor, date_)`: builds candidate start times from that counsellor's `working_hours` for the given weekday (`SESSION_LENGTH_MINUTES=50`, `SLOT_STEP_MINUTES=60`, `BOOKING_WINDOW_DAYS=21` rolling horizon, `MIN_LEAD_TIME=2h`), excludes anything covered by an existing confirmed `Booking` row, and excludes anything from `calendar_sync.get_busy_intervals` (empty today). Also holds `SESSION_FEE_DISPLAY`, a cosmetic string shown in the booking summary — not persisted, no payment processing exists.
  - `core/calendar_sync.py` — the only seam where a real Google/Outlook Calendar integration would plug in. `get_busy_intervals()`/`create_calendar_event()` both check `CalendarAccount.is_connected` and log-and-no-op today; there is no live OAuth flow and no counsellor-facing "connect your calendar" UI (counsellors aren't site users). Making this real means: registering an OAuth app with the provider, building a connect flow (likely admin-only, since there's no counsellor login), and replacing the bodies of these two functions — nothing else in the codebase should need to change.
  - `core/models.py` — content models: `ServiceCategory` (slug/name/eyebrow/intro/order — one row per service page, also the grouping for that service's topic checklist), `Topic` (slug/label/`category` FK/order — one specialty checkbox), `Counsellor` (slug — the lookup key used everywhere, e.g. `?counsellor=<slug>`, kept read-only in admin after creation since renaming it would orphan `Booking.counsellor_slug` history/bookmarked links; name/credentials/`photo` `ImageField` blank=True + `photo_placeholder` fallback string/location/`languages`/`modes`/`modalities` as `JSONField(default=list, blank=True)` lists/intro/bio/`fee_note` (reference-only, not read by scheduling.py or the booking page — real fees vary a lot per counsellor but wiring per-counsellor pricing into the booking flow is a separate follow-up)/`user` — nullable `OneToOneField` to the Django user model, groundwork for a future counsellor self-edit login, unpopulated today), `CounsellorWorkingHours` (counsellor FK/weekday 0=Mon..6=Sun/start_time/end_time — one row per working window, multiple rows per day support split shifts), `CounsellorSpecialty` (counsellor FK/topic FK/`level` — `IntegerChoices` PRIMARY=2/SECONDARY=1 — the primary/secondary specialty pick per topic). Transactional models: `Booking` (`counsellor_slug` as a plain `CharField`, **still not a FK** against `Counsellor` even though that model now exists — converting it would touch the double-booking constraint, `scheduling.py`'s query and `admin.py`'s `list_filter` for no behavior change, so it's deferred until there's an actual reason, e.g. the self-edit portal; `client_name`/`client_email`/`mode`/`start_at`/`end_at`/`status`/`calendar_event_id`/`created_at`) and `CalendarAccount` (stub — `counsellor_slug`, `provider`, `is_connected`, OAuth token fields, all unpopulated in this pass). The double-booking guard is a `UniqueConstraint` on `(counsellor_slug, start_at)` conditioned on `status="confirmed"` — SQLite has no useful row-level locking, so this constraint (not `select_for_update`) is what actually prevents two concurrent bookings for the same slot; the loser's `INSERT` raises `IntegrityError`, caught in `views.book`.
  - `core/migrations/0002_...py` (schema for the 5 content models) and `0003_seed_service_and_counsellor_data.py` (a `RunPython` data migration that seeds the topic taxonomy + all 7 real counsellor profiles — content hardcoded directly in the migration file, not imported from `core/data.py`, since migrations must stay replayable even after the app's data-access code changes; has a working `reverse_seed_data` so `migrate core 0002` cleanly undoes the seed).
  - `core/admin.py` — registers `ServiceCategory`/`Topic` plainly (staff/superuser-only in practice). Registers `Counsellor` with `CounsellorWorkingHoursInline`/`CounsellorSpecialtyInline` (`TabularInline`s — this is today's UI for the primary/secondary specialty picker; a real checkbox-grid self-edit UI is a future follow-up, not built yet) and a permission restriction so a non-superuser tied to a `Counsellor` via `.user` can only ever see/edit their own row (`get_queryset` filters the changelist, `has_change_permission`/`has_view_permission` check `obj.user_id == request.user.id`, `has_add_permission`/`has_delete_permission` are superuser-only) — no real counsellor `User` accounts exist yet, this is groundwork for when they do. Also registers `Booking` (list/filter by date/counsellor/mode/status, searchable) and `CalendarAccount`; together with `Counsellor` this is the only staff UI for the site's DB-backed content.
  - `core/templates/base.html` — shared shell (head, header/nav, footer, `{% static 'js/index.js' %}`) with `title`/`meta_description`/`extra_css`/`extra_js`/`content` blocks. All other templates extend this. Header/footer "Book a session" links point at `{% url 'book' %}`.
  - `core/templates/index.html` — homepage content block: hero (with the portrait carousel and a "Book a first session" CTA linking to `book`), the four service cards (each an `<a>` linking to `service_detail` by slug), approach, a "Meet our counsellors" teaser card linking to `counsellors`, and the CTA marquee (also linking to `book`). **This is the file to edit for homepage content/markup changes; edit the static CSS/JS files directly for style/behavior changes.**
  - `core/templates/service_detail.html` — per-service page: intro, a `topic-chip` button grid built from `topics`, a mode filter (Either/Online/In person), a matches section, and a "See all counsellors" link to the directory. Counsellor data reaches the client via `{{ counsellors|json_script:"counsellors-data" }}` (a `<script type="application/json">` tag), not inline JS — `static/js/service.js` reads and `JSON.parse`s it, and each match card's CTA links to `book` with that counsellor preselected (`?counsellor=<slug>`).
  - `core/templates/counsellors.html` — the directory: one `<li>` per counsellor, each a native `<details>`/`<summary>` accordion (no JS required — keyboard/ARIA semantics come free from the element). The `<summary>` shows photo/name/designation/intro and an always-visible "Book with [Name]" link to `book?counsellor=<slug>` (safe to nest inside `<summary>` since it navigates rather than toggling); expanding reveals the full bio, specialty tags, modalities, and languages/location/mode.
  - `core/templates/booking.html` — the booking page: a counsellor picker (cards styled like `.match-card`), a calendar + slot-chip availability section (populated via `static/js/booking.js` fetching `booking_availability`), a mode toggle limited to the selected counsellor's `modes`, name/email fields, and a summary panel — one real `<form method="post">` to `book`, with hidden inputs (`counsellor_slug`/`date`/`time`/`mode`) kept in sync by JS so the actual submit is a normal page navigation, not a fetch-driven SPA flow. Counsellor data (including `working_hours`, needed client-side to grey out non-working days) reaches JS the same `json_script` way as `service_detail.html`.
  - `core/templates/booking_confirmed.html` — static confirmation page for a given `Booking`.
- `static/css/index.css`, `static/js/index.js` — global styles/behavior shared by every page (scroll-reveal, hero carousel) via `base.html`. The homepage counsellors teaser's "View" circle (`.view-badge`) is a plain always-visible link/badge with a simple hover scale-up — no cursor-follow gimmick (an earlier mouse-following variant looked out of place on mobile and was removed).
- `static/css/service.css`, `static/js/service.js` — additive, loaded only by `service_detail.html` via the `extra_css`/`extra_js` blocks. `service.js` builds a `topic slug → label` map from the rendered chips (no data passed for this beyond the DOM), tracks up to 5 selected topics + one mode filter in memory, scores each counsellor as `sum(specialties[topic] for topic in selected) + (1 if mode matches)`, and renders the top 3 counsellors with score > 0. No network requests — everything after initial page load is client-side over the embedded JSON.
- `static/css/counsellors.css` — additive, loaded only by `counsellors.html`; styles the accordion rows and chevron (rotated/faded via the native `[open]` attribute selector, no JS).
- `static/css/booking.css`, `static/js/booking.js` — additive, loaded only by `booking.html`. `booking.js` renders the counsellor cards and a custom month-calendar widget entirely client-side from the embedded JSON (greying out weekdays the selected counsellor doesn't work and dates outside the `BOOKING_WINDOW_DAYS` horizon), then does one `fetch('/book/availability/?counsellor=...&date=...')` per date selection to render real slot chips — the server's `get_available_slots` is always the authority on what's actually bookable, the client-side calendar greying is just a UX nicety.
- `static/images/face_*.png` — portrait photos, reused in the homepage hero's `.hero-faces` carousel (`#facesTrack` in `index.html`), the homepage teaser's preview avatars, and as the default `photo_placeholder` fallback for each seeded `Counsellor` row (see `core/migrations/0003_seed_service_and_counsellor_data.py`). In the hero carousel, the track is duplicated (each image listed twice) for a seamless CSS `translateX` loop; `static/js/index.js` runs a `requestAnimationFrame` loop that scales each photo based on its live distance from the viewport centre (small at the edges, large in the middle) — adding/removing images there means updating both copies in the track markup.
- `db.sqlite3` — dev SQLite database. Schema: `Booking`/`CalendarAccount` (`0001_initial.py`), `ServiceCategory`/`Topic`/`Counsellor`/`CounsellorWorkingHours`/`CounsellorSpecialty` (`0002_...py`, seeded by `0003_seed_service_and_counsellor_data.py`) plus Django's own auth/admin/sessions tables.

## Settings notes

`ProjectCounsellingSite/settings.py` is mostly the stock `startproject` output and has not been hardened: `DEBUG = True`, a placeholder `SECRET_KEY`, and `ALLOWED_HOSTS = []`. Treat these as dev-only; flag them if asked to prepare this for production.

`STATICFILES_DIRS = [BASE_DIR / 'static']` is required in `settings.py` for `runserver` to discover the project-root `static/` folder — it isn't inside an app, so Django's default app-static autodiscovery won't find it without this.

`MEDIA_URL`/`MEDIA_ROOT` (`BASE_DIR / 'media'`) back `Counsellor.photo` uploads. `ProjectCounsellingSite/urls.py` serves `MEDIA_ROOT` itself when `DEBUG=True` (via `django.conf.urls.static.static`) — fine for dev, but a real deployment needs a proper media host/storage backend instead.

`TIME_ZONE = 'Asia/Kolkata'` (changed from the generated default of `'UTC'`) — every counsellor is India-based and `core/data.py`'s `working_hours` are authored as India-local wall-clock strings (e.g. `"09:00"`); with `USE_TZ=True`, `timezone.make_aware(...)` anchors naive datetimes to this zone, so leaving it at `UTC` would silently shift every counsellor's advertised hours by 5.5 hours.
